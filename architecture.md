# Architecture

## 1. Style: Modular Monolith, Microservices-Ready

We build **one deployable FastAPI application**, internally split into strict, decoupled modules (Python packages) that communicate through well-defined interfaces and an internal event bus. This gives startup-speed now, and any module (Payments, Delivery, Chat) can be extracted into its own service later **without changing its public contract**, because:

- Each module owns its own DB tables (no cross-module raw SQL joins)
- Cross-module communication happens via **services (in-process) or events**, never direct repository/session access
- Modules never import another module's internal (non-exported) files

```
apps/
  backend/               → FastAPI modular monolith (Phase 1)
  frontend/               → HTML/CSS/JavaScript customer/seller/admin web app (no framework/bundler required)
  mobile/                 → Flutter app (Phase 2)
```

There is no shared-types code-generation package in this stack (that pattern exists to keep TypeScript frontends in sync with a TS backend). Instead, the FastAPI-generated OpenAPI schema (`/openapi.json`) is the single source of truth the frontend JS can reference manually when building `fetch` calls.

## 2. Tech Stack (Phase 1)

| Layer | Choice | Why |
|---|---|---|
| API framework | **FastAPI** (Python) | async-first, Pydantic validation built in, auto OpenAPI docs, maps cleanly to layered/DDD-style modules |
| API style | REST + OpenAPI (auto-generated by FastAPI) | GraphQL deferred — one API style until there's a proven need |
| Realtime | **FastAPI native WebSockets** | chat, order/rider tracking, notifications — no separate Socket.IO server needed |
| DB | **PostgreSQL** | relational integrity for orders/payments/wallets |
| ORM / Migrations | **SQLAlchemy 2.0 (async)** + **Alembic** | typed models, explicit reviewable migration history |
| Cache/Queue | Redis + **Celery** (or RQ for a lighter footprint) | rider matching, notification jobs, rate limiting, background payout processing |
| Object storage | **Cloudinary** | product images, shop logos, chat attachments, verification docs — stores `secure_url` + `public_id` per asset |
| Search | Postgres full-text (`tsvector`) | swap for Meilisearch when catalog > ~100k items |
| Maps | Mapbox | rider tracking, geocoding, route ETA |
| Auth | JWT (access+refresh) via `python-jose`/`PyJWT` + `passlib` (bcrypt/argon2) | email/password + phone OTP; OAuth added Phase 2 |
| Payments | MTN MoMo API, Stripe | mobile money + card, behind a `PaymentProvider` interface |
| Frontend | **Plain HTML + CSS + JavaScript** (ES modules, no build step required) | server-rendered-simple pages + fetch-driven interactivity; keeps the stack small and dependency-free |
| State/data | Native `fetch` + small hand-rolled JS modules (`api/`, `state/`) | no React Query/Zustand — a thin `apiClient.js` wraps fetch, a small pub-sub or plain module-level state holds UI state |
| Mobile | Flutter | Phase 2 |
| CI/CD | GitHub Actions → Docker → single managed host (Fly.io/Render/DO App Platform) | Kubernetes deferred until traffic justifies it |
| Monitoring | Sentry (errors) + basic Grafana Cloud/Logtail (logs) | Prometheus/self-hosted Grafana deferred |

## 3. Module Map (Phase 1)

```
backend/app/
  auth/            registration, login, OTP, JWT, RBAC dependencies
  users/           profile, roles, addresses
  catalog/         products, categories, shops
  orders/          cart, checkout, order state machine
  payments/        wallet ledger, MoMo, Stripe, payouts
  delivery/        rider matching, delivery state machine, geolocation
  chat/            conversations, messages (FastAPI WebSocket routes)
  notifications/   in-app + push (FCM) fan-out
  reviews/         product/seller/rider ratings
  admin/           moderation, dispute queue, dashboards
  common/          dependencies, middleware, exception handlers, schemas
  events/          internal event bus (typed domain events as Pydantic models)
  core/            settings/config, database session, security utilities
```

Cross-module rule of thumb: `orders` module never touches `payments` tables directly — it emits `OrderPaid`/`OrderDelivered` events; `payments` and `delivery` subscribe to what they need.

## 4. Core Domain Event Flow (the one unique workflow)

```
Customer checkout
  → OrderCreated (orders)
  → payments.reserve_funds() [escrow hold]
  → OrderPaid (payments)
  → seller notified, clicks "Request Rider"
  → RiderRequested (delivery)
  → delivery.match_nearest_riders() [Redis geo query, radius expand]
  → push notification to N nearest riders
  → first Accept wins → RiderAssigned
  → status pipeline: EN_ROUTE_TO_SELLER → PICKED_UP → EN_ROUTE_TO_CUSTOMER → DELIVERED
  → customer confirms (or auto-confirm after timeout) → DeliveryConfirmed
  → payments.release_funds() → seller payout + rider payout queued (Celery task)
  → both parties prompted to rate
```

All state transitions are enforced by explicit state machines (not free-form status strings) in `orders` and `delivery` modules — this is the single most important invariant in the whole system, since money and physical goods depend on it.

## 5. Architectural Principles

- **Layered architecture per module**: `domain/` (entities, business rules, no framework deps — plain Python classes/dataclasses) → `application/` (use-case/service classes) → `infrastructure/` (SQLAlchemy repositories, external API clients: Cloudinary, MoMo, Stripe, Mapbox) → `interface/` (FastAPI routers, Pydantic request/response schemas)
- **Repository pattern**: routers never build SQLAlchemy queries directly — they call an application service, which calls a repository
- **CQRS only where it earns its keep**: order/delivery state machines and analytics read models; not applied dogmatically everywhere
- **Idempotency**: all payment-mutating endpoints and webhook handlers require an idempotency key
- **Event-driven for cross-cutting concerns**: payments, notifications, and delivery dispatch are all consumers of domain events, not direct callers of each other
- **Every external integration (payment provider, maps, SMS, Cloudinary) sits behind an interface** in `infrastructure/`, so providers can be swapped/added (this is how MoMo → +Orange Money, or Mapbox → Google Maps, or Cloudinary → S3-compatible storage, happens later without touching business logic)

## 6. Environments

`local` (Docker Compose: Postgres, Redis, FastAPI backend, static frontend server) → `staging` → `production`. Same Docker images promoted through environments; config via env vars only (see `security.md`). Cloudinary credentials (cloud name, API key/secret) are env vars, never committed.

## 7. Deferred Architecture Decisions (Phase 2+)

- Split `delivery` and `payments` into standalone services once delivery volume needs independent scaling
- Introduce GraphQL gateway if mobile team needs flexible querying
- Introduce Elasticsearch/Meilisearch when catalog search quality becomes a complaint, not before
- Kubernetes migration once traffic > single-host capacity
- Add a frontend build step (bundler, TypeScript, or a framework) only if the vanilla JS approach starts causing real maintenance pain — not preemptively
